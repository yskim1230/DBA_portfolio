# DBA Portfolio (On-Premise / Cloud) — 김영석

천재교육 DBA로서 **전사 핵심 DB 운영(온프레미스)**과 **클라우드 전환(통합/이관 PoC)** 과정에서 수행한 업무를  
**증빙 중심(Runbook/스크립트/로그/정량 지표)** 으로 정리한 포트폴리오입니다.

---

## Portfolio Structure

- **On-Premise**: IDC 기반 운영(가용성/성능/보안/백업/복구/모니터링)
- **Cloud**: NCP 기반 클라우드 운영(통합 1안 PoC, DB 설정 표준화, 덤프/복구 자동화, 접근통제/트러블슈팅)

---

## 1. On-Premise (IDC)

### 1.1 인프라 요약
- 전사 핵심 DB(MSSQL) 및 서비스별 MySQL/MariaDB 운영
- All-Flash 스토리지 + dHCI 기반 가상화 인프라
- 백업/소산/DR 정책 운영 및 정기 복구 검증

### 1.2 고가용성(HA) / 장애 대응
- MySQL/MariaDB **MHA 기반 자동 Failover**
- 장애 대응 Runbook 및 복구 리허설(정기 테스트 포함)

### 1.3 Backup & Recovery
- Full/Incr/일간/주간/월간 등 다층 백업 체계
- 소산 백업 및 보관 정책(ISMS 기준) 운영
- 실제 복구/최신화 수행 내역 기반 절차서

### 1.4 성능/운영 최적화
- 인덱스/쿼리 튜닝, 리빌드, 실행계획 기반 개선
- 용량/로그성 테이블 관리 정책, 운영 자동화 스크립트

### 1.5 보안/감사(ISMS)
- 계정/권한 정책, 접근제어, 마스킹 등 보안 통제
- ISMS 증적/감사 대응 문서화(재현 가능한 절차 중심)

### 1.6 모니터링/관측(Observability)
- MaxGauge / PMM 기반 실시간 모니터링
- SMTP 알람 연계 및 장애 조기 탐지 체계

## 2. Cloud (NCP)

### 2.1 클라우드 전환 배경 및 목표

* 온프레/분산 구성에서 NCP Cloud DB for MySQL 중심으로 운영 전환 과정 참여
* 통합 운영을 문서/절차/로그(증적)**로 남기는 방향으로 정리
* (비용 관점 포함) 자원 인벤토리 -> 과금 요인 분해 -> 통합/정리(Dev/Stg 포함)로 이어지는 실행 단위로 작업

### 2.2 자원 인벤토리 구축 및 운영 기준 수립

* Cloud DB 인스턴스/DB 목록 인벤토리화(표준 컬럼/Owner/용도/환경/접속정보/백업 정책 등)
* 자원 통합/정리 판단을 위한 기준값(중복 DB, 유휴, 통합 대상, 보관정책 등) 정의
* 증적: 인벤토리 문서/스프레드시트, 운영 기준(정의서), 갱신 이력

### 2.3 네트워크/접근통제 운영 (VPC 기반)

* VPC/Subnet/ACG/NACL 구조 파악 후 접근 통제 기준 정리
* DB 접속 장애를 네트워크(경로/ACL) vs 권한(계정/GRANT) vs 인증(SSL/엔드포인트)로 분해해 트러블슈팅
* 운영 관점에서 VPN 접근/엔드포인트/방화벽 정책을 문서화

### 2.4 DB 운영 설정 표준화 (MySQL 8.4 기준)

* 파라미터/정책을 개인 설정이 아니라 베이스라인(표준 설정)으로 정리

  * 예: lower_case_table_names, charset/collation, sql_mode 등
* 운영 리스크 항목을 주의가 아니라 정책 항목으로 고정

  * 예: event_scheduler 사용 기준, lock timeout 기준, slow log 수집 기준, 세션/쿼리 관측 기준
* 증적: 표준 설정 체크리스트, 변경 이력, 적용 전/후 비교 캡처

### 2.5 **분산 DB 통합 이관 파이프라인 PoC** (Dump/Restore)

* 분산 DB를 표준 네이밍 규칙(예: 'lms_lrm_eng01')으로 재구성하는 통합안 기반 PoC 수행
* 단순 dump/restore가 아니라, Schema-first 덤프(--no-data) → Import → 검증 로그 생성으로 재실행 가능한 절차를 만듦
* 정량 로그(테이블 수/완결성 체크)로 남김
* 증적: export/import 스크립트, 실행 로그, 검증 결과(txt), 네이밍 규칙/통합 기준 문서

### 2.6 관측/성능 운영 체계 구축 (PMM + 슬로우쿼리)

* PMM 기반으로 모니터링 PoC -> 운영 적용까지 연결 (대시보드/지표 확인 체계 포함)
* Active Session 3초 샘플링 / Long-running 세션 확인 등 운영자가 바로 쓰는 관측 단위로 정리
* 슬로우 쿼리 로그를 일일 수집/정렬/Top N 추출하는 루틴을 문서화 및 스크립트화
* 증적: PMM 대시보드 캡처, 수집 스크립트(active_3s 등), 일일 리포트/산출물(txt)

### 2.7 백업/배치/운영 자동화(서버 작업 포함)

* 작업 서버 기준 크론잡 기반 운영 작업(백업/정리/수집) 파악 및 개선 포인트 도출
* 보관/삭제(rolling) 기준을 두고 용량 리스크(디스크 풀) 방지 관점에서 운영 정리
* 증적: crontab 내역, 배치 스크립트, 보관정책 문서/런북

### 2.8 스키마/메타데이터 거버넌스 (EXERD + 변경관리)

* EXERD 기반 통합 ERD 갱신 및 컬럼 변경 반영
* 용어사전/물리/논리 스키마 일관성을 유지하면서 변경관리(요청>반영>검증) 흐름 정리
* (사례형) 외부 요건(Keris 등)으로 인한 컬럼 길이 변경/스키마 변경 대응을 절차화
* 증적: 변경관리 문서, EXERD 변경 내역, 반영 검증 캡처

---

## 운영 원칙 (공통)
- **도메인 로직은 앱, DB는 무결성과 저장에 집중**
- 변경은 항상 **검증(정량 지표/로그) + 롤백 전략**을 포함
- 자격증명/민감정보는 문서에 미기재(Secret 분리)

---

## How to Read
1) On-Premise에서 운영 기본기(백업/복구/HA/보안/튜닝)를 확인  
2) Cloud에서 전환 과정(통합/이관 PoC, 설정 표준화, 접근통제, 자동화)을 확인  
3) 각 문서는 “배경 → 절차 → 검증 → 결과/리스크” 구조로 작성
